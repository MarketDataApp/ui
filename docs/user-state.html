Overview

The user module provides shared user state management for *.marketdata.app. It decouples user data fetching from any specific UI component so that multiple components on the same page can share a single user state without redundant API calls.

  • Stale-while-revalidate with a 1-minute TTL — returns cached data instantly, refreshes in the background only when stale.
  • Request deduplication — concurrent calls share a single in-flight request.
  • Subscriber pattern — components subscribe to state changes (including logout) via onUserChange().

API Reference

Function Returns Description
fetchUser(options?) Promise<Object|null> Fetch the current user. Returns cached data within TTL; revalidates in background when stale. Pass { apiUrl } to override the endpoint.
onUserChange(callback) () => void Subscribe to user state changes. Callback receives the new user object or null on logout. Returns an unsubscribe function.
_clearCache() void Reset all caches and state. For testing only.

Caching Behavior

fetchUser() resolves through three tiers before hitting the network:

  1. Memory cache — module-level variable (fastest, current page session)
  2. sessionStorage — key marketdata_user (persists across same-tab navigations)
  3. Network fetch — GET /api/user/ with credentials: 'include'

TTL (1 minute)

After a successful fetch, the result is considered fresh for 60 seconds. Within the TTL, fetchUser() returns cached data immediately with zero network activity. After the TTL expires, it returns the stale cache and kicks off a background revalidation.

Request Deduplication

If a fetch is already in flight, additional calls return the same promise instead of starting a new request. Three components calling fetchUser() simultaneously result in a single network request. 

fetchUser()
  |
  +-- cache hit?
  |     +-- fresh (< 60s) --> return cached (no network)
  |     +-- stale (>= 60s) --> return cached + revalidate in background
  |
  +-- cache miss --> await network fetch (deduplicated)
        +-- success --> cache result, notify subscribers if data changed
        +-- failure --> retry up to 2x with exponential backoff

Subscriber Pattern

onUserChange(callback) lets any component react to user state changes without polling or re-fetching.

When subscribers are notified

  • A background revalidation returns different data than what was cached (user profile updated)
  • A background revalidation returns an error after previously having a cached user (logout detected — callback receives null)

When subscribers are NOT notified

  • Cache is within TTL (no revalidation happens at all)
  • Background revalidation returns the same data
import { fetchUser, onUserChange } from '@marketdataapp/ui/user';

// Subscribe to changes
const unsub = onUserChange((user) => {
  if (user) {
    console.log('User updated:', user.name);
  } else {
    console.log('User logged out');
    showLoginButton();
  }
});

// Fetch user (subscribers notified if data changes)
const user = await fetchUser();

// Later: clean up
unsub();

User Profile Component

The user-profile component is the first consumer of the user state module. It renders a login button (logged out) or Gravatar avatar with optional dropdown menu (logged in). Internally it calls fetchUser() and subscribes via onUserChange() for logout detection.

Import from @marketdataapp/ui/user-profile. 

import { initUserProfile } from '@marketdataapp/ui/user-profile';

const cleanup = await initUserProfile({
  container: document.getElementById('my-avatar'),
  dropdown: true,          // enable dropdown menu
  // loginUrl, logoutUrl, dashboardUrl, etc. — all customizable
});

// cleanup() removes DOM and unsubscribes from user state
